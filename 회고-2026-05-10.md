## 1. 오늘 한 것 요약

### 시스템 흐름이 한 바퀴 돌게 됨

**원장(미나님) 영역**
- 회원가입(비번 확인) / 로그인 / 매장 만들기
- 대시보드 (실시간 통계: 오늘 / 이번 주 / 확인 대기)
- 시술 메뉴 CRUD (6개 카테고리 항상 노출)
- 예약 관리 (확정/취소/완료/노쇼/되돌리기 + 입금 토글 + 동의서 뱃지)
- 고객 목록 (검색/정렬/직접 추가)
- 고객 차트 (상담/진단 분리, 칩, 펜 시안, 시술 타임라인)
- 매장 설정 (PIN 설정·변경, 영업시간 변경)

**손님 영역**
- 매장 페이지 (메뉴 보기 + 예약하기 + 내 예약 조회)
- 예약 폼 (달력 + 30분 슬롯 + 시간 충돌 자동 차단 + 시술 범위 시각화)
- 예약 완료 페이지 (입금 안내)
- 본인 예약 조회 (폰번호 기반)

**고객 모드 (iPad에서 손님 직접 입력)**
- 피부 타입 (큰 버튼)
- 알러지/특이사항 (칩 다중선택 + 펜 그림)
- 원하는 디자인 (눈썹/입술/속눈썹 카테고리별 칩 + 펜 그림)
- 원하는 컬러 (눈썹/입술 컬러 칩)
- 시술 동의서 (메뉴 카테고리별 자동 템플릿 + 동의 체크 + 전자서명)
- 백버튼/스와이프 보호 — 빠져나오려면 직원 PIN 입력

### DB 스키마 (sql 11개 적용)
- `salons` `profiles` `menus` `bookings` `customers` `consents`
- View: `booking_slots` (개인정보 X, 충돌체크용)
- RPC: `get_booking_by_token`, `get_bookings_by_phone`
- Triggers: 예약 시 customer 자동 등록 / 시술 완료 시 통계+리터치일 갱신 / customer_number 자동 부여

## 2. 다음 세션 시작점

### 🔴 1순위: 고객 모드 펜 그림이 차트 페이지에 안 보임
- 고객 모드에서 손님이 알러지/디자인을 펜으로 그린 게 DB(`customers.allergies_drawing`, `design_drawing`)에 저장됨
- 그런데 미나님이 보는 차트 페이지(`/dashboard/customers/[id]`)에서는 그 그림이 표시 안 됨
- 차트 페이지의 알러지 / 디자인 섹션에 손님이 그린 펜 이미지를 read-only로 보여주는 작업 필요
- 작업 분량 작음 (30분 내)

### 🔴 2순위: 시술 전/후 사진
- 7대 차별 기능 1·2번 (시안 페이지 + 전/후 비교 슬라이더)의 핵심
- Supabase Storage 버킷 셋업 + RLS 정책 + 업로드 UI + 갤러리 + 비교 슬라이더
- 작업 분량 큼 (2~3시간)

### 🟡 그 외 후보
- 고객 정보 수정/삭제 (이름·전화·생일 직접 수정, 삭제)
- 차트에서 바로 다음 예약 잡기 버튼
- 고객 등급 자동 (방문횟수 기준 VIP/단골/신규)
- 자유 태그 (단골, 신부, 민감 등)

### 🟢 후순위
- SMS 알림 (solapi 매장별 API 키 입력형 — 결정됨)
- 동의서 PDF 다운로드 (한글 폰트 임베딩 필요)
- 다음 리터치 자동 알림 (SMS 의존)

### 다시 시작할 때 명령어 예시
- "고객 모드 펜 그림 차트에 표시해줘" → 1순위
- "사진 기능 만들자" → 2순위
- "어디까지 했지?" → 이 파일 + CLAUDE.md 보고 정리

## 4. 잘된 부분 (예상보다 잘됨)

### 운영에 진짜 필요한 디테일을 잡음
1. **고객 모드의 보안성**: PIN + 백 버튼 보호 → 손님이 iPad 들고 다른 거 보지 못하게 막음. 단순 솔루션이지만 매장 운영의 실제 페인 포인트
2. **iPad/Apple Pencil 우선 설계**: 펜 입력 캔버스 — 콜라보살롱은 펜 입력 자체가 없음. 시술자가 시안 직접 그릴 수 있는 게 큰 차별
3. **상담/진단의 명확한 분리**: 손님이 본인이 더 잘 아는 정보(피부 타입, 알러지)는 직접 입력, 원장은 진단에 집중. 워크플로우 자체가 자연스러움
4. **시간 충돌 자동 차단**: 메뉴 duration_minutes 만큼 점유, 같은 날 다른 예약과 겹치면 회색 + "예약됨". 영업시간 넘는 슬롯도 자동 비활성. 콜라보살롱보다 깊이 있음
5. **고객번호 + 동명이인 구분**: 매장별 #1, #2 자동 — 단순한데 실제 운영에 매우 도움
6. **카테고리별 동의서 템플릿**: 눈썹/입술/속눈썹 메뉴 따라 자동으로 다른 동의서 → 반영구 전용 SaaS다움이 처음으로 보임
7. **안전한 권한 구조**: anon SELECT 정책 닫고 view + RPC로 필요한 데이터만 노출. 손님이 다른 손님 정보 못 봄

## 5. 부족한 부분 (개발자 시점)

### A. 디자인 / UX

1. **로딩 스피너 없음** — "불러오는 중..." 텍스트만. 스켈레톤 UI 도입 필요
2. **에러 처리 단순함** — 빨간 글씨 또는 alert만. 토스트/스낵바 같은 일관 시스템 없음
3. **빈 상태(empty state) 톤 일관성** — 메시지마다 약간씩 다름
4. **마이크로 인터랙션 부족** — 카드 호버, 상태 변경 애니메이션 적음
5. **다크 모드 없음** — 라이트 톤 하나만 (반영구 매장은 보통 분위기 있는 조명이라 다크모드 수요 있을 수 있음)
6. **접근성 미점검** — aria-label, 키보드 내비게이션, 색상 대비
7. **모바일 폰 화면 일부 어색** — iPad 우선이라 작은 폰에선 버튼 비율이 큼
8. **이미지 최적화 없음** — Next/Image 미사용 (사진 들어가면 필요)

### B. 기능

1. **검색이 단순** — `includes` 매칭만. 한국어 초성 검색, 자모 분리 검색 없음 (네이버 예약은 됨)
2. **벌크 액션 없음** — 여러 예약 한 번에 확정/취소/메시지 등
3. **CSV/엑셀 내보내기 없음** — 매출 분석, 고객 명단 외부 사용 불가
4. **이미지 업로드 없음** — 가장 큰 차별 기능이 아직
5. **SMS/카톡 알림 없음** — 실 운영 거의 필수
6. **푸시 알림 없음** — 새 예약 들어와도 미나님 즉시 인지 불가 (브라우저 닫으면 모름)
7. **실시간 동기화 없음** — Supabase Realtime 미사용. 다른 기기 변경 시 새로고침 필요
8. **결제/구독 없음** — 모든 기능 무료 상태, 매출 모델 미연결
9. **분석 대시보드 빈약** — 카운트만. 매출, 메뉴별 인기도, 재방문율 X
10. **다중 직원 X** — 1인 매장 가정. 여러 디자이너 매장 미대응
11. **요일별 영업시간 / 휴무일 X** — 단일 시작/종료 시간만
12. **예약금 PG 결제 X** — 계좌입금 안내만. 실제 입금/환불 자동화 없음
13. **고객 차트 버전관리 X** — 시술마다 차트 스냅샷이 없음 (변화 추적 불가)
14. **사진 위 그리기 X** — 시안 페이지 핵심. 손님 얼굴 사진 위에 디자인 그리기

### C. 콜라보살롱 등 통합형 vs BrowChart

**우리가 더 나은 점**
- 반영구 전용 깊이 (카테고리별 동의서, 디자인 칩, 펜 시안)
- iPad/Apple Pencil 우선 설계
- 고객 모드 보안 (PIN)
- 시간 충돌 자동 차단의 깊이

**그쪽이 나은 점**
- 무료 (BrowChart는 추후 유료)
- 통합형 — 미용실/네일 한 앱에서
- SMS/카톡 발송 빌트인
- 결제 PG 빌트인
- 고객 모바일 앱 (네이티브)
- 매출 통계 다양함
- 직원 권한 관리
- 미국 진출 → 다국어
- 마케팅 자료/도움말 풍부

### D. 네이버/카카오 예약 vs BrowChart
- 네이버는 검색 노출이 강력 (마케팅) — 우리는 자체 링크만
- 카톡 채널 연동 (단골 관리) — 우리 X
- 알림톡 — 우리 X
- 다만 네이버/카카오는 차트나 동의서가 거의 없는 단순 예약 도구 — 우리는 차트가 핵심

### E. 코드 / 엔지니어링

1. **테스트 없음** — 단위 테스트, E2E 테스트 X. 릴리스 전 수동 QA만
2. **에러 모니터링 없음** — Sentry 등 미연결. 프로덕션 에러 추적 불가
3. **Analytics 없음** — GA4, Mixpanel 등 없음. 사용자 행동 데이터 없음
4. **CI/CD 미설정** — 수동 배포
5. **백엔드 로직 분리 부분적** — Supabase 트리거/RPC로 일부 처리하지만, SMS 발송 큐 같은 건 Edge Function 필요해질 것
6. **타입 안전성 일부 약함** — `as unknown as` 캐스트 일부 있음
7. **컴포넌트 라이브러리 없음** — 재사용 컴포넌트 적음. 같은 모달/입력 스타일이 여러 곳에서 복사됨
8. **i18n 없음** — 한국어 하드코딩
9. **PIN 평문 저장** — 매우 짧은 4~6자리라 위험은 낮지만 hash 권장
10. **서명/그림이 base64로 DB에 직접 저장됨** — 큰 데이터는 Storage로 옮기는 게 좋음 (지금은 작아서 OK)

## 결론

**MVP 1차 핵심은 거의 다 됨.** 손님 예약부터 차트, 동의서, 고객 모드까지 한 바퀴 돈다.

**가장 큰 미구현은 사진** (시안 + 전/후 비교). 이거 끝나면 7대 차별 기능 절반이 진짜로 채워짐.

**그 다음 큰 미구현은 SMS** (실 운영 필수). 솔라피 방향은 결정됨 (B안 매장별 키).

다음 세션에서 1순위(펜 그림 차트 표시 — 빠른 마무리) → 2순위(사진) 순으로 가면 좋아요.
